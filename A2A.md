# A2A.md - Katala A2A協業プロトコル

**目的**: しろくま × カニ のDiscord A2A協業における運用ルールとセキュリティ基準

---

## 1. 役割分担

| 役割 | 担当 | 責務 |
|------|------|------|
| **ホスト** | しろくま | 設計判断 / レビュー / マージ権限 / セキュリティ監査 |
| **実装** | カニ | Issue起票 / コーディング / テスト / サブエージェント実行 |
| **承認** | ニコラス | Go/No-Go判断 |

---

## 2. 通信ルール

### 2.1 チャンネル
- `#dev-katala` - 一般開発会話
- `#dev-katala-a2a` - A2A bot間通信専用

### 2.2 メンション位置
**原則: メンションは返信の最後に置く**
- Discordの分割送信対策（最初のメッセージだけ見る問題防止）
- 例: 「内容説明...<@user_id>」

### 2.3 クールダウン
- 同一セッション内での連続応答: **10秒間隔**
- 自己メッセージは無視

### 2.4 1メッセージ1レス
- 分割投稿の場合、最後のメッセージにメンション
- 同一 `message_id` は1回のみ処理

## 3. 開発フロー

```
Issue起票
    ↓
ブランチ作成: feature/a2a-{番号}-{概要}
    ↓
実装 + テスト
    ↓
PR作成（相互レビュー）
    ↓
ニコラス承認
    ↓
マージ → Issueクローズ
```

### 3.1 ブランチ命名規則
- `feature/a2a-{issue番号}-{簡潔な概要}`
- 例: `feature/a2a-1-webauthn-assertion`

### 3.2 PRテンプレート
- 概要（Issueリンク）
- 変更内容
- テスト結果
- レビュー依頼先

## 4. エラー共有プロトコル

### 4.1 エラー発生時
**原因3択を提示**:
1. 設定問題（OpenClaw config等）
2. コード問題（実装バグ等）
3. 環境問題（外部サービス等）

### 4.2 ログ項目（固定）
```
message_id: {id}
error: {エラー内容}
drop_reason: {原因分類}
timestamp: {ISO8601}
```

### 4.3 エスカレーション基準
- **同一エラー3回連続** → 方針転換検討
- **方針転換後も失敗** → 報告・承認待ち

## 5. 知見の蓄積

### 5.1 教訓のドキュメント化
- 気づきは即座に記録
- `knowledge/` または `memory/` に反映

### 5.2 設定変更の記録
- 必ずcommit + 理由を記述
- 例: `fix(config): allowBots=true for A2A (#reason)`

### 5.3 監査ログ化
重要操作は「原因/変更/結果」を3行で残す:
```
原因: Webhook未承認でメンション検出不可
変更: allowFromにWebhook ID追加
結果: メール→Discord通知が機能
```

## 6. セキュリティ原則

### 6.1 基本原則（8項目）

| # | 原則 | 実践 |
|---|------|------|
| 1 | **秘密はチャットに置かない** | Webhook/token/keyは貼らない。貼ったら即ローテーション |
| 2 | **最小権限** | allowlist方式（openではなくallowlist）、ID限定、`requireMention=true` |
| 3 | **公開/内部分離** | インフラ詳細（IP/SSH/内部URL）は限定チャネル |
| 4 | **Secrets配布は"相手に渡さない"** | GitHub UI/CLIで本人が直接設定 |
| 5 | **失敗時標準対応** | ①露出停止 ②ローテーション ③影響確認 ④再発防止 |
| 6 | **A2A特有防御** | メンション末尾固定、1msg1res、自己msg無視、クールダウン |
| 7 | **設定とコードの優先順位** | 動かない時は**まず設定を疑う** |
| 8 | **OpenClaw実装ルール** | destructive操作は確認、変更はcommit+理由、ログは構造化 |

### 6.2 セキュリティホールと対策（追加8項目）

| # | ホール | 対策 |
|---|--------|------|
| 9 | Webhook過剰権限 | 用途別分離、漏れたら即再発行 |
| 10 | GitHub Actionsのsecret漏洩 | `set -x`禁止、ログにsecret出さない、外部PRでsecret使わない |
| 11 | PRからの設定改変 | `.github/workflows/*`をCODEOWNERSで承認必須 |
| 12 | A2A無限ループ/誤反応 | 末尾メンション+同一message_id 1回処理 |
| 13 | DMセッション分離誤解 | シークレットはDMでも渡さず、直接UI/CLI入力 |
| 14 | 公開チャンネルでの内部情報露出 | 要約のみ公開、具体パス・値は限定チャネル |
| 15 | "temporary例外"の恒久化 | 例外設定に期限をつける+戻し確認 |
| 16 | モデル/ツール切替時の監査不足 | 「原因/変更/結果」を3行で残す |

### 6.3 追加考慮事項

| # | 項目 | 理由 |
|---|------|------|
| 17 | **Token有効期限管理** | 長期間有効なトークンは漏洩リスク高い。定期的ローテーション |
| 18 | **監査ログ保持期間** | いつまでログを残すか決めてない。長期保存はリスク、短期は追跡不可 |
| 19 | **プロンプトインジェクション** | 外部メール・メッセージをモデルに渡す際、悪意のある指示が混入する可能性 |
| 20 | **クロスセッション情報漏洩** | Discordグループ（公開的）とDM（私的）の境界。個人情報が誤って公開チャンネルに |
| 21 | **自動化ジョブのサイレント失敗** | 通知設定漏れで失敗に気づかない。必ず通知先を明示 |

**優先対応**: 11 (CODEOWNERS)、15 (期限付き例外)、20 (セッション境界)、21 (通知設定)

## 8. コンテキスト管理

### 8.1 閾値管理
- **40%**: proactive整理の検討開始
- **50%**: `/compact` or `/reset` 実行

### 8.2 メモリ構造（3層）
1. **日次メモ** (`memory/YYYY-MM-DD.md`) - 生ログ
2. **長期記憶** (`MEMORY.md`) - 蒸留された重要事項
3. **構造化知識** (`knowledge/`) - トピック別ナレッジ

### 8.3 忘却ルール
- 古い日次メモは**90日で `archive/` に退避**
- MEMORY.mdに昇格してない = 重要じゃなかった、と割り切る
